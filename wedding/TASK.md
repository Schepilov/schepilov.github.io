# **ТЗ: сайт-приглашение на свадьбу с персональными ссылками и сбором RSVP**

## **1\. Общая задача**

Разработать свадебный сайт-приглашение на внешнем хостинге и домене заказчика с механикой персональных приглашений по уникальным ссылкам.

Сайт должен:

- показывать гостям информацию о приглашении;
- в зависимости от типа приглашения показывать один или два блока с датой, временем и местом;
- позволять подтвердить или отклонить участие для каждого отдельного приглашённого;
- собирать данные по каждому гостю внутри приглашения;
- поддерживать заполнение одной формы за всю семью / пару / домохозяйство («группу»);
- в отдельных случаях позволять одиночному гостю добавить +1;
- сохранять ответы в Google Sheets;
- не раскрывать гостям список других приглашённых;
- быть свёрстан по наработкам макета в Figma (взять из него цвета, стили, шрифты, часть графики);
- корректно работать на разных типах устройств;
- не индексироваться поисковыми системами, но корректно отображаться в превью ссылки в мессенджерах и соцсетях.

## **2\. Бизнес-логика приглашений**

### **2.1. Типы приглашений**

Каждое приглашение относится к одной из групп:

- приглашение на **оба дня**;
- приглашение **только на второй день**.

### **2.2. Единица приглашения**

Персональная ссылка создаётся не обязательно на одного человека, а на сущность:

- семья;
- пара;
- домохозяйство;
- отдельный гость.

Одна ссылка = одно приглашение = одна группа людей, для которых заполняется форма.

### **2.3. Состав приглашения**

Для каждого приглашения заранее задаётся список ФИ гостей.

Гость, открывший ссылку:

- не должен искать себя в общей базе;
- должен видеть только тех людей, которые входят в его приглашение.

### **2.4. Заполнение за группу**

Форма должна позволять одному человеку заполнить данные сразу за всех людей внутри приглашения.

Это обязательное требование для удобства:

- семей;
- пар;
- пожилых гостей.

### **2.5. Возможность убрать гостя из состава ответа**

Для гостей, заранее включённых в приглашение, должна быть возможность не подтверждать участие.

Важно:

- исходный состав приглашения в базе не удаляется физически;
- "убрать гостя из списка" в пользовательском интерфейсе должно означать, что этот человек не участвует в финальном ответе либо отмечен как неактуальный/не присутствует;
- первичные данные приглашения должны сохраняться.

Конкретная реализация допустима в одном из двух вариантов:

- человек остаётся в списке, но получает статус «не смогу присутствовать»;
- человек помечается служебным флагом как исключённый из ответа, без удаления из исходной базы.

Предпочтительно использовать вариант 1 как основной.

### **2.6. Возможность добавить +1**

Функция добавления +1 должна работать только для тех приглашений, где это разрешено.

Условия:

- +1 разрешён только для одиночных гостей;
- если в приглашении уже несколько заранее заданных людей, функция +1 по умолчанию недоступна;
- для +1 должна появляться отдельная карточка с полями, аналогичными обычному гостю.

## **3\. Ограничения и принципы безопасности**

### **3.1. Запрет на общий поиск по базе гостей**

На сайте не должно быть:

- общего поиска по ФИ;
- автокомплита по всему списку гостей;
- возможности узнать состав других приглашений;
- возможности перебором проверить, кто приглашён.

### **3.2. Доступ только по персональной ссылке**

Каждая группа гостей получает ссылку с уникальным токеном.

Пример:

- <https://domain.ru/i/8fK2mQpR>
- <https://domain.ru/i/H7x9Ld2A>

Требования к токену:

- случайный;
- трудно угадываемый;
- не содержать ФИ, ID по порядку и других очевидных идентификаторов.

### **3.3. Изоляция данных**

По токену должно возвращаться только содержимое конкретного приглашения:

- тип приглашения;
- состав приглашённых внутри него;
- флаг возможности +1;
- уже сохранённые ответы, если они существуют.

Никакие данные по другим приглашениям на клиент не выдаются.

## **4\. Архитектура решения**

### **4.1. Общая архитектура**

Сайт размещается на внешнем хостинге на домене заказчика.

Хранение данных:

- Google Sheets.

Рекомендуемая схема интеграции:

- фронтенд сайта на внешнем хостинге;
- тонкий серверный/API-слой для чтения и записи данных в Google Sheets.

### **4.2. Варианты реализации API**

Допустимые варианты:

- **Apps Script Web App** как прослойка между сайтом и Google Sheets;
- **serverless API** (например, Cloudflare Workers / Vercel Functions / Netlify Functions) с доступом к Google Sheets API.

Предпочтение:

- если нужна максимальная простота и быстрый запуск - Apps Script;
- если нужна более чистая инженерная архитектура - serverless-функции.

### **4.3. Что нельзя делать**

Не допускается:

- прямой доступ фронтенда к Google Sheets без серверной прослойки;
- хранение или раздача клиенту полной базы гостей;
- публичный API без проверки токена приглашения.

## **5\. Дизайн и макеты**

### **5.1. Источник дизайна**

Макет сайта готовится самостоятельно на основе наработок в **Figma**: взять из него цвета, стили, шрифты, часть графики. Дополнить недостающие состояния

### **5.2. Адаптивность макета**

Макет должен быть адаптивным и предусматривать отдельные состояния/поведение для следующих типов экранов:

- **узкие смартфоны**, включая старые или небольшие устройства, что особенно важно для пожилых гостей;
- **обычные современные смартфоны**;
- **планшеты**;
- **компьютеры и ноутбуки**;
- широкие десктопные экраны в форматах **16:9** и **21:9**.

### **5.3. Требование к вёрстке по адаптивности**

При вёрстке необходимо обеспечить:

- корректное отображение всех блоков на всех указанных разрешениях;
- отсутствие горизонтального скролла там, где он не предусмотрен дизайном;
- читаемость текста без необходимости увеличения страницы;
- удобное нажатие на кнопки и элементы формы на тач-устройствах;
- сохранение визуальной логики макета при переходе между брейкпоинтами.

## **6\. Пользовательские сценарии**

### **6.1. Сценарий: семья / пара / домохозяйство**

- Пользователь открывает персональную ссылку.
- Видит приветственный экран приглашения.
- Видит один или два блока по дням свадьбы в зависимости от типа приглашения.
- Переходит к анкете.
- Видит список людей, входящих в приглашение.
- Заполняет данные по каждому человеку.
- Отправляет один общий ответ.

### **6.2. Сценарий: одиночный гость без +1**

- Пользователь открывает персональную ссылку.
- Видит свои данные.
- Заполняет анкету только на себя.
- Отправляет ответ.

### **6.3. Сценарий: одиночный гость с +1**

- Пользователь открывает персональную ссылку.
- Видит себя как основного гостя.
- При необходимости нажимает кнопку «Добавить +1».
- Появляется дополнительная карточка для +1.
- Пользователь заполняет данные по себе и по +1.
- Отправляет ответ.

### **6.4. Повторное открытие ссылки**

Если ответ уже был сохранён, при повторном открытии ссылки сайт должен уметь показывать экран «ответ уже получен» с возможностью отредактировать ранее введённый ответ.

## **8\. Поля анкеты**

Для каждого гостя необходимо собирать следующие данные.

### **8.1. Фамилия имя**

- для заранее заданных гостей - отображается предзаполненным;
- для +1 - вводится вручную.

### **8.2. Присутствие**

Варианты ответа:

- смогу присутствовать;
- не смогу присутствовать;
- сообщу позже.

Поле обязательное. Но нужно адаптировать текст, потому что сейчас тексты написаны так, будто каждый сам за себя отвечает, но многие будут заполнять сразу за всю группу, поэтому и формулировки должны быть адаптированы под это

### **8.3. Предпочтения по напиткам**

Должны быть доступны варианты:

- безалкогольные;
- белое вино;
- красное вино;
- игристое;
- пиво;
- крепкое.

Требование:

- это должен быть множественный выбор, если разработчик не видит серьёзных UX-противоречий;
- если выбран вариант крепкое, должно раскрываться дополнительное текстовое поле с ручным вводом:
  - уточни, какое именно.

### **8.4. Аллергии / непереносимости**

Выбор из двух опций:

- Нет аллергий и непереносимостей в еде
- Есть аллергии и непереносимости в еде

Выбор второй опции добавляем два инпута с мульти-выбором вариантов:

- аллергии
- непереносимости

Нужно подготовить список наиболее часто встречающихся в России пищевых аллергенов и непереносимостей

Также должна быть возможность указать свой вариант ответа, если нужного варианта нет в выпадающем списке

### **8.5. Предпочтения по горячему**

Два варианта на выбор:

- мясо (скрытая подсказка: говяжьи щёчки с луком порей и кремом пармезан)
- рыба (скрытая подсказка: лосось с спаржей и цветной капустой)

## **9\. Логика работы с данными гостей**

### **9.1. Предзаполненные гости**

Для гостей, заранее включённых в приглашение:

- фамилии и имена выводятся автоматически;
- пользователь заполняет данные по каждому.

### **9.2. Исключение гостя из ответа**

Если пользователь не подтверждает участие конкретного гостя:

- запись не удаляется из исходной базы;
- в ответе сохраняется статус или флаг, отражающий отсутствие участия.

### **9.3. +1**

Для добавленного +1 должны сохраняться:

- фамилия имя;
- признак, что это добавленный гость;
- все остальные данные анкеты.

## **10\. Анимации, загрузка и стабильность интерфейса**

### **10.1. Общие требования**

При вёрстке должны быть реализованы аккуратные и визуально приятные состояния загрузки и подгрузки данных.

Цель:

- избежать "скачков" интерфейса;
- сделать восприятие сайта более плавным и качественным;
- скрыть технические задержки загрузки данных.

### **10.2. Лоадер**

На начальной загрузке сайта или ключевых экранов должен использоваться лоадер.

Требования:

- визуально соответствовать стилю макета;
- не выглядеть стандартным браузерным спиннером без стилизации;
- не ломать композицию страницы.

### **10.3. Скелетоны**

Для экранов, где данные подгружаются асинхронно, необходимо использовать скелетоны.

Например:

- карточки гостей;
- блоки с датой и местом;
- анкета при подгрузке данных по токену.

Требования:

- скелетоны должны повторять примерную геометрию итогового контента;
- скелетоны должны предотвращать скачки вёрстки;
- после загрузки контента переход от скелетона к данным должен быть плавным.

### **10.4. Плавность интерфейса**

Допускаются и приветствуются:

- мягкие fade-in / fade-up анимации;
- плавное появление карточек;
- анимации раскрытия дополнительных полей, например поля для уточнения крепкого алкоголя;
- плавное добавление карточки +1.

Не допускаются:

- навязчивые, тяжёлые или слишком долгие анимации;
- анимации, мешающие пожилым пользователям быстро пройти форму;
- резкие прыжки блоков при загрузке или валидации.

## **11\. Структура данных**

Ниже описана рекомендуемая логическая структура. Реальные названия листов и колонок могут отличаться, но смысл должен быть сохранён.

### **11.1. Технические листы**

Технические листы нужны для корректной работы системы. Они не предназначены для передачи организаторам в исходном виде.

#### **Лист invites**

Одна строка = одно приглашение.

Рекомендуемые поля:

- invite_token
- invite_label - название группы, например "Семья Ерунковых"
- invite_type - both_days / day2_only
- plus_one_allowed - true/false
- max_extra_guests - обычно 0 или 1
- status - active/inactive
- created_at

#### **Лист invite_members**

Одна строка = один заранее заданный гость внутри приглашения.

Рекомендуемые поля:

- member_id
- invite_token
- full_name

## **12\. Представление данных для организаторов**

### **12.1. Общее требование**

Организаторы не должны работать с техническими таблицами в сыром виде.

Необходимо предусмотреть отдельные человекочитаемые таблицы/листы для передачи организаторам.

### **12.2. Принцип**

Система должна разделять:

- техническое хранение данных;
- человеческое представление данных.

### **12.3. Рекомендуемые организаторские листы**

#### **Лист Организаторам - гости**

Одна строка = один гость.

Пример человекочитаемых колонок:

- **Группа**
- **Гость**
- **Тип приглашения**
- **Присутствие**
- **Предпочтения по напиткам**
- **Крепкий алкоголь**
- **Аллергии / непереносимости**
- **Горячее блюдо**
- **Дата и время ответа**

### **12.4. Требование к формату колонок**

В "человеческих" листах:

- использовать русские понятные названия колонок;
- не использовать технические идентификаторы в качестве основных пользовательских колонок;
- напитки выводить не набором технических флагов, а одной читаемой строкой, например:
  - Безалкогольные
  - Красное вино, игристое
  - Пиво, крепкое

### **12.5. Способ формирования организаторских листов**

Допустимы 2 подхода:

- отдельные листы, собираемые из технических данных формулами или скриптом;
- генерация отдельных агрегированных строк при записи ответа.

Предпочтительный вариант:

- организаторские листы строятся на основе технических данных, без дублирования логики вручную.

## **13\. API: минимальные требования**

### **13.1. Получение данных приглашения**

Нужен endpoint для получения данных приглашения по токену.

Он должен возвращать:

- валидность приглашения;
- тип приглашения;
- список гостей внутри приглашения;
- разрешён ли +1;
- ранее сохранённые ответы, если есть.

### **13.2. Сохранение ответа**

Нужен endpoint для отправки формы.

Он должен:

- валидировать токен приглашения;
- валидировать структуру ответа;
- сохранять или обновлять ответы в Google Sheets;
- возвращать успешный или ошибочный статус.

### **13.3. Обновление ответа**

Повторная отправка по тому же токену должна:

- либо обновлять текущий набор ответов;
- либо создавать новую ревизию ответа.

Предпочтительно:

- сохранять ревизионность или хотя бы дату последнего обновления.

## **14\. SEO, индексирование и превью ссылок**

### **14.1. Запрет на индексирование**

Сайт не должен индексироваться поисковыми системами.

Требования:

- закрыть сайт от индексирования на уровне мета-тегов и/или других доступных механизмов;
- страницы приглашений не должны попадать в поисковую выдачу;
- поисковые роботы не должны использовать страницы сайта как обычные индексируемые страницы.

### **14.2. Поддержка сниппетов и превью ссылок**

Несмотря на запрет индексации, сайт должен корректно поддерживать мета-данные для предпросмотра ссылки в мессенджерах и соцсетях.

Должны поддерживаться:

- заголовок страницы;
- описание;
- превью-картинка.

### **14.3. Требования к мета-данным**

Необходимо предусмотреть:

- базовые &lt;title&gt; и meta description;
- Open Graph мета-теги;
- при необходимости Twitter/X card мета-теги.

### **14.4. Требования к превью**

При отправке ссылки в мессенджерах и соцсетях сайт должен по возможности формировать красивый сниппет:

- заголовок;
- описание;
- изображение.

При этом содержание сниппета не должно раскрывать приватные персональные данные гостя сверх согласованного объёма.

## **15\. Валидации**

Обязательные проверки:

- токен приглашения существует и активен (если нет токена, то скрываем блок анкеты гостя)
- нельзя отправить ответ по несуществующему приглашению;
- нельзя добавить +1, если это запрещено;
- по каждому отображённому гостю должен быть указан статус присутствия;
- если выбрано крепкое, требовать уточняющий текст;
- ФИ для +1 обязательно к заполнению;
- количество +1 не должно превышать разрешённое значение.

## **16\. Нефункциональные требования**

### **16.1. Простота для пожилых пользователей**

Интерфейс должен быть максимально простым:

- без поиска себя по базе;
- без необходимости вручную вводить ФИ для уже заданных гостей;
- с понятными подписями;
- с крупными, очевидными элементами интерфейса.

### **16.2. Мобильная версия**

Сайт должен быть адаптирован под смартфоны, так как значительная часть гостей будет открывать ссылку из мессенджеров.

### **16.3. Скорость и простота**

Форма должна открываться быстро и не требовать регистрации, входа, авторизации или дополнительных действий от гостя.

### **16.4. Визуальная стабильность**

При загрузке и подгрузке данных интерфейс не должен заметно "скакать".

Необходимо минимизировать layout shift за счёт:

- скелетонов;
- резервирования места под контент;
- плавных переходов между состояниями.

## **17\. Результат разработки**

В результате должен быть реализован рабочий свадебный сайт-приглашение, который:

- открывается по персональной ссылке;
- корректно адаптируется под узкие телефоны, обычные смартфоны, планшеты, десктопы 16:9 и широкие экраны 21:9;

- позволяет в нужных случаях добавить +1;
- сохраняет ответы в Google Sheets;
- формирует отдельное человекочитаемое представление данных для организаторов;
- использует красивые состояния загрузки, лоадер и скелетоны;
- не раскрывает список других гостей;
- не индексируется поисковыми системами;
- корректно отображает сниппет ссылки в соцсетях и мессенджерах
